昨日串联：昨天理解了 Ingress Controller 如何将“域名路由”声明式自动化。今天向前一步，看 Ingress Controller 自身如何做到高可用。

今日精读：Pod 的容忍 (Toleration) 与节点亲和性 (Node Affinity)

Ingress Controller 也是跑在 Pod 里的。如果一个集群有 10 个 Node，如何保证 Ingress Controller Pod 只会调度到特定的 3 个节点上？这用到两个核心调度策略。

1. 容忍 (Toleration) —— “可以在哪”

英语原文 (K8s 官方文档摘录)

A toleration is a property of a Pod that allows (but does not require) the Pod to schedule onto nodes with matching taints.

Taints are the opposite — they allow a node to repel a set of pods.

Taints and tolerations work together to ensure that pods are not scheduled onto inappropriate nodes. One or more taints are applied to a node; this marks that the node should not accept any pods that do not tolerate the taints.

中文翻译

容忍是 Pod 的一个属性，它允许（但不强制）Pod 调度到带有匹配污点的节点上。

污点则相反——它允许节点排斥一组 Pod。

污点和容忍协同工作，确保 Pod 不会被调度到不合适的节点。节点被施加一个或多个污点，这标记了该节点不应接受任何不容忍这些污点的 Pod。

通俗理解：

概念 类比
污点 (Taint) 节点上贴的标签：“这个房间只有戴防毒面具的人能进”
容忍 (Toleration) Pod 声明的属性：“我戴了防毒面具，我可以进那个房间”

我的 mini-apiserver 场景：假设你的阿里云 ECS 上运行着《密盾》和 mini-apiserver 两个服务。你可以给节点打一个污点 service=prod:NoSchedule，然后让《密盾》的 Pod 带对应容忍，而 mini-apiserver 不带——这样 mini-apiserver 就不会被调度到这台“生产”节点上，实现业务隔离。

2. 节点亲和性 (Node Affinity) —— “更想去哪”

简单版（硬性要求）：

```yaml
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
      - matchExpressions:
        - key: kubernetes.io/os
          operator: In
          values:
          - linux
```

这个规则是硬约束：Pod 只能调度到 os=linux 的节点。如果没有符合条件的节点，Pod 将一直处于 Pending 状态，直到有匹配的节点出现。

更实用的版本（软性偏好）：

```yaml
affinity:
  nodeAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      preference:
        matchExpressions:
        - key: node-type
          operator: In
          values:
          - ssd
```

这个规则是软偏好：Pod 更想去有 SSD 的节点，但如果 SSD 节点资源不够，它也可以调度到普通节点。

底层本质：
Scheduler 调度 Pod 时分两阶段——Filter（过滤不可行节点）和 Score（给剩余节点打分）。污点/容忍属于 Filter 阶段（不满足直接排除），软亲和性属于 Score 阶段（加分项，不满足也还能调度）。

设计哲学：
与《密盾》三权分立的解耦策略一致——Node 只负责声明“我的能力/限制”（污点），Pod 只负责声明“我的需求/偏好”（容忍/亲和），调度器在中间协调。各司其职，互不侵入。

3. Pod 中断预算 (PodDisruptionBudget, PDB)——主动维护时的“安全保障”

英文原文 (K8s 官方文档摘录)

A PodDisruptionBudget (PDB) is a policy that limits the number of Pods of a replicated application that are down simultaneously from voluntary disruptions. For example, a quorum-based application would like to ensure that the number of replicas running is never brought below the number needed for a quorum.

中文翻译

Pod 中断预算 (PDB) 是一种策略，它限制了在自愿性中断（如节点维护、kubectl drain）中同时不可用的 Pod 数量。例如，一个基于仲裁的应用程序希望确保运行的副本数永远不会低于法定人数所需的数量。

通俗理解：
当你需要主动维护集群（比如升级节点、kubectl drain）时，K8s 会安全地驱逐 Pod。PDB 就是告诉 K8s：“不管你怎么驱逐，我的服务最少要保持 1 个副本在线。低于这个数就拒绝驱逐。”

与昨日内容的串联：
昨天理解了 Ingress Controller 自动管理 Nginx。但如果 10 台 Node 里某台需要维护，K8s 如何保证 Ingress Controller 不会“全军覆没”？靠的正是 PDB。

最小可用性保护示例：

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: ingress-pdb
spec:
  minAvailable: 1
  selector:
    matchLabels:
      app: ingress-nginx
```

这个 PDB 保护的是带有 app: ingress-nginx 标签的 Pod，要求任何主动驱逐操作必须确保至少 1 个 Pod 保持运行。如果主动驱逐会导致所有 Pod 被同时杀掉，这个操作就会被 K8s 暂时阻止。

关联你的实践：
昨天你给 mini-apiserver 的部署流程是：

```
备份旧版 → 停旧进程 → 启新进程
```

这是人工保证“服务不中断”的原始方式。在 K8s 里，PDB + Ingress Controller 的组合，是在自动化层面实现了等价效果——K8s 不会同时杀掉所有 Pod，确保新 Pod 已经起来并接受流量后，才继续驱逐其他 Pod。这种滚动更新的策略，和你的手动备份 → 替换 → 启动，本质上是同一个目标的不同实现层级。

把人工的运维经验，提炼成云原生的理解——这正是把部署经验转化为体系化知识的表现。
