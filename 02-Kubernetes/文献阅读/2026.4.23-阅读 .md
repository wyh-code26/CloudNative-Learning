精读主题：Service——Pod 的稳定访问入口

今日主题直接对应《密盾》架构图中 Nginx → Go API → ZKP 模块的网络连通设计。

---

英语原文（K8s 官方文档摘录）

Service

An abstract way to expose an application running on a set of Pods as a network service.

With Kubernetes you don't need to modify your application to use an unfamiliar service discovery mechanism. Kubernetes gives Pods their own IP addresses and a single DNS name for a set of Pods, and can load-balance across them.

Motivation

Kubernetes Pods are created and destroyed to match the desired state of your cluster. Pods are nonpermanent resources. If you use a Deployment to run your app, it can create and destroy Pods dynamically.

Each Pod gets its own IP address, however in a Deployment, the set of Pods running in one moment in time could be different from the set of Pods running a moment later.

This leads to a problem: if some set of Pods (call them "backends") provides functionality to other Pods (call them "frontends") inside your cluster, how do the frontends find out and keep track of which IP address to connect to? What about the parts of the frontend that are outside the cluster?

中文翻译

Service（服务）

一组 Pod 上运行的应用程序的抽象网络服务暴露方式。

使用 Kubernetes，你无需修改应用程序来使用陌生的服务发现机制。Kubernetes 为每个 Pod 分配独立 IP，为一组 Pod 分配单一 DNS 名称，并可在其间进行负载均衡。

设计动机

Pod 会随集群期望状态不断创建和销毁，本质上是非永久资源。如果你用 Deployment 运行应用，它会动态地创建和销毁 Pod。

每个 Pod 有独立 IP，但某一时刻运行的 Pod 集合与下一秒运行的集合可能完全不同。

这就引出问题：如果一组 Pod（"后端"）为其他 Pod（"前端"）提供功能，前端如何发现并跟踪应连接哪个 IP？集群外的前端部分又该怎么办？

K8s 精读笔记

1. Service 的本质

Service 解决的核心问题：Pod IP 会变，但调用方需要一个不变的地址。

概念 映射到传统网络
Pod IP 动态 DHCP 分配的 IP，Pod 重启就变
Service ClusterIP 静态 VIP（虚拟 IP），创建后不变
Endpoints Service 到 Pod IP 的映射表，由 kube-proxy 实时维护
kube-proxy 类似 L4 反向代理，用 iptables/IPVS 将流量从 VIP 转发到 Pod IP

底层本质：Service 不是一个真实运行的服务进程，而是 iptables/IPVS 规则 + CoreDNS 中的一条 A 记录。当你访问 Service 的 ClusterIP 时，数据包在 Linux 内核网络栈中被 kube-proxy 的规则拦截并 DNAT 到后端 Pod IP。

2. Service 类型

类型 用途 访问范围
ClusterIP 集群内部通信（默认） 仅集群内
NodePort 在每台 Node 上开一个端口 公网/内网可访问 NodeIP:Port
LoadBalancer 云厂商提供外部 LB 公网，需云支持
ExternalName 将 Service 映射到 CNAME 集群内通过 DNS 访问外部服务

3. Service 如何工作（数据流）

```
用户请求 → ClusterIP:Port
                │
                v
        kube-proxy (iptables/IPVS 规则)
                │
                ├──→ Pod A (10.244.1.5:8082)
                ├──→ Pod B (10.244.1.6:8082)
                └──→ Pod C (10.244.2.3:8082)
```

每步详解：

1. DNS 解析：Pod 内访问 my-svc.default.svc.cluster.local，CoreDNS 返回 ClusterIP
2. 数据包到达 ClusterIP，内核 netfilter 钩子触发 iptables 规则
3. kube-proxy 的规则随机/轮询选择一个 Pod IP，执行 DNAT
4. 响应包原路返回，对调用方透明

4. 与《密盾》项目的直接关联

现在的架构：

```
用户浏览器 → HTTPS → Nginx:443 → proxy_pass http://go-api:8090
```

若迁移到 K8s，架构演变为：

```yaml
# 密盾后端 Service（ClusterIP）
apiVersion: v1
kind: Service
metadata:
  name: midun-api
spec:
  selector:
    app: midun-api          # 选中所有带此标签的 Pod
  ports:
  - port: 8082              # Service 对外端口
    targetPort: 8082        # Pod 内进程监听端口
```

实际效果：

· 该 Service 获得一个固定 ClusterIP（如 10.96.0.2），即使 Pod 重启、漂移到其他节点，IP 不变
· Nginx 配置可从 proxy_pass http://8.210.229.103:8090 变为 proxy_pass http://midun-api.default.svc.cluster.local:8090
· 这意味着：Nginx 零依赖服务器公网 IP，所有组件间通信全走内部 DNS，安全性和可维护性飞跃

设计哲学：解耦“服务身份”（Service Name/IP）与“服务实例”（Pod），与《密盾》的“凭证持有者→签发方→验证方”三解耦设计思路完全同构。

今日 K8s 小结

维度 理解
Service 是什么 Pod 的稳定前端 + 负载均衡器，由 iptables/DNS 实现，非真实进程
为何需要 Pod IP 随销毁重建而变，调用方需要不变地址
核心机制 Label Selector 选 Pod → Endpoints 维护映射 → kube-proxy 写 iptables 规则 → CoreDNS 注册域名
映射之前所学 Nginx upstream 块做后端池 = K8s Service 做 Pod 池，原理一样，但 Service 是声明式自动化的
密盾关联 未来将 go-api:8082 抽象为 Service，Nginx 直连 Service DNS 名即可，消除 IP 硬编码
