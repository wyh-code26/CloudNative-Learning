今日精读：Ingress — 集群的“智能反向代理”

与今晚反复折腾的 Nginx 反代和域名路由直接相关。Ingress 就是 K8s 内部帮你自动配置 Nginx/Traefik 的那层。

1. 英语原文与翻译 (来自 Kubernetes 官方文档)

Ingress
An API object that manages external access to the services in a cluster, typically HTTP. Ingress may provide load balancing, SSL termination and name-based virtual hosting.
Ingress
一个管理集群内服务外部访问的 API 对象，通常用于 HTTP(S)。Ingress 可以提供负载均衡、SSL/TLS 终结（HTTPS）和基于域名的虚拟主机（同一端口多个域名）。

2. 核心概念拆解

· Ingress 是什么？：它不是一个跑着的服务进程，而是一组路由规则（像“如果访问 api.wuyuhangcn.com/v1，就转发给 midun-api 服务”）。
· Ingress Controller 是什么？：真正干活的东西。它是运行在集群里的 Nginx (或 Traefik, HAProxy) 实例，负责读取你写的 Ingress 规则并执行流量转发。
· 为什么需要它？：没有 Ingress，每个 HTTP 服务都要申请一个公网 LoadBalancer，成本高且难管理。Ingress 让你用一个入口 (8.210.229.103) 对外，根据不同域名/路径分流给后端不同服务，零额外公网端口。

3. 🔗 与《密盾》明早部署的直接关联

明早要做的事情是：
用户浏览器 → HTTPS → Nginx:443 → proxy_pass http://go-api:8090

在 K8s 中，这个流程被抽象为：

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: midun-ingress
spec:
  # 🚀 明早我们要做的 HTTPS 证书，在这里只需引用一个 Secret 名字即可搞定
  tls:
  - hosts:
    - zkp.wuyuhangcn.com
    secretName: midun-tls
  rules:
  - host: zkp.wuyuhangcn.com  # 你的子域名
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: midun-api    # 对应你今晚打包的容器
            port:
              number: 8090
```

· 本质：今晚用命令行配的 Nginx 规则，在云原生架构下被这个 YAML 声明式替代了。
· 设计哲学：K8s 让你只关注“我要什么”（域名、路径、后端服务），具体怎么配 Nginx 由 Controller 自动完成。和你《密盾》的“三权分立”同一种解耦思维。
