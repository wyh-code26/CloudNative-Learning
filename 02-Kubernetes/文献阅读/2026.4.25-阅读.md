今日精读：Ingress Controller — 集群的“自动化 Nginx 管理员”

今晚手动修改 Nginx 配置、重载服务、排查 443 端口缺失的问题——整个过程耗时约 40 分钟。在 K8s 里，Ingress Controller 用声明式配置自动完成这些操作。

Ingress 是什么？Ingress Controller 是什么？

概念 类比今晚的操作 说明
Ingress 你脑子里的“我想让 zkp.wuyuhangcn.com 指向 8090 端口” 一组路由规则 YAML，描述“什么域名走什么服务”
Ingress Controller 你今晚手动执行的 nginx -t && systemctl reload 真正干活的 Nginx/Traefik 实例，读取 Ingress 规则并自动更新自己的配置
Secret /etc/letsencrypt/live/... 下的证书文件 存储 TLS 证书的 K8s 对象

今晚的操作 vs Ingress Controller 自动化

```
今晚的手动流程：
  修改 sites-available/zkp 文件
  → nginx -t 检查语法
  → systemctl reload nginx 热重载
  → curl 验证结果
  → 发现报错 → 看日志 → 再改 → 再重载

Ingress Controller 自动化：
  kubectl apply -f ingress.yaml
  → Controller 监听到新 Ingress 对象
  → 自动生成 Nginx 配置
  → 自动执行 nginx -t（语法检查）
  → 自动执行 reload（热重载）
  → 自动更新证书（通过 cert-manager）
```

一个 Ingress YAML 对应你今晚写的全部 Nginx 配置

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: midun-ingress
spec:
  tls:
  - hosts:
    - zkp.wuyuhangcn.com
    secretName: zkp-tls          # 等于你今晚的 ssl_certificate 那几行
  rules:
  - host: zkp.wuyuhangcn.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: midun-api      # 等于你今晚的 proxy_pass http://127.0.0.1:8090
            port:
              number: 8090
```

底层本质：Ingress Controller 本质是一个持续运行的控制器循环，它用 K8s API 监听 Ingress 对象变化 → 翻译成 Nginx 配置 → 写入配置文件 → 执行热重载。这个循环替代了今晚手动执行的“改配置 → 测语法 → 重载”三步操作。

设计哲学：K8s 把这个循环自动化后，只需要声明“我要什么”（哪个域名、哪个后端），Controller 负责“怎么做”（怎么写 Nginx 配置、什么时候重载）。和《密盾》的“三权分立”解耦思路同构——分离关注点，让专业组件做专业事。

---

今日小结：今晚用 sed + nginx -t + systemctl reload 手动完成了 Nginx 配置的查找、替换、验证、热重载全流程。在 K8s 里，这套流程被 Ingress Controller 自动化了，但底层操作完全一样。当以后写 Ingress YAML 时，心里知道它背后在做的事情——就是今晚亲手做过的事情。
