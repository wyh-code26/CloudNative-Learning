命令：head
 
- 高频场景：快速查看文件前几行、管道截断输出、检查文件开头格式
- 踩坑点：直接写  head 5  不生效，必须带  -n ；管道截断会出现 Broken pipe
- 底层发生了什么：顺序读取文件开头数据，读到指定行数后立即关闭文件
- 映射到系统：典型的顺序读、按需读取，类似只加载配置文件头部校验
 
命令：tail
 
- 高频场景：查看日志末尾、取最新记录、快速定位最近报错
- 踩坑点： tail -n +6  是从第6行开始显示，容易和取最后6行搞混
- 底层发生了什么：通过  lseek  跳到文件尾部，反向扫描换行符定位行
- 映射到系统：基于偏移随机读取，只读取文件末尾部分，大文件依然高效
 
命令：tail -f
 
- 高频场景：实时追踪日志文件
- 踩坑点：日志轮转后  tail -f  会失效，需改用  tail -F （按文件名追踪）
- 底层发生了什么：调用  inotify_add_watch()  注册文件描述符，内核在文件变化时通知用户进程
- 映射到K8s：Controller 的 Informer 通过 Watch API 监听资源变化，等价于分布式  tail -f 
 
命令：tail -F
 
- 高频场景：生产环境追踪会被 logrotate 切割的日志文件
- 踩坑点：文件被删除重建后  tail -f  不再更新， -F  才能自动重连
- 底层发生了什么：持续检测文件 inode 变化，文件重新生成后自动重新打开
- 映射到K8s：对应 Informer 断开重连 + 重新 List&Watch，保证监听不中断
